Roadmap

The roadmap and milestones define the project direction and priorities.

As developers on a hackable, flexible Vim, we're drawn to evermore new ideas; but the relevance and impact of the project hinges on our ability to balance exploration of the New with execution of the Now. If a feature doesn't advance the current milestone, it may be postponed.

Current

0.2

  • GUI
  • Windows support
  • Pre-packaged "RelWithDebInfo" build for Windows
  • Pre-built binaries for non-Windows platforms (tentative)

0.3

  • Lua host, lua becomes a runtime dependency, shipped with the core editor
  • Basic support for LSP et al.

Meta

There are release milestones and meta milestones ("buckets").

  • Version-number milestones (0.1, 0.2, 0.3, â€¦) assign issues to production releases.
    • The nearest upcoming version is assigned a target date.
    • Low-risk fixes in the current branch are first committed to the stable branch, then merged to master. For example, if the current stable release is 0.1.1 and a bug-fix is assigned to the 0.2 milestone, it does not mean users need to wait until 0.2 to get the fix. The patch will be committed to the stable branch and released in 0.1.2.
  • todo is a bucket for triaged tickets which should be on the roadmap but cannot yet be assigned to a concrete milestone.
  • unplanned is a bucket for low-priority items or items lacking a feasible path to completion.

Completed

0.1

The primary goal of the 0.1 release ("first public release") is to reach a stable, functional baseline that can be built upon and used as a benchmark for further progress. Releasing a stable product to a larger audience will increase feedback on potential "blind spots" and will signal the most effective direction for the project.
